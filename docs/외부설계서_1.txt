Program Data Flow 기본 설계서
5차 수정 설계안 | 2026-04-09 11:52
========================================

[용어 정의]
- RDB: On-line remote Firebase DB
- LDB: Off-line local Excel file DB
- DB는 Event들을 단순 저장하며 Event의 날짜와 시각 정보가 포함되어 있다.
- App: 설계 대상 Program을 지칭


[기본 전제]
- Event는 각각 Event ID를 갖고 ID는 Event 발행 날짜와 시각 정보가 포함된다. (Event를 발행하는 곳은 모두 시각 동기화가 보장됨)
- App은 여러 PC에서 구동되어 서로 RDB를 통해 데이터 동기화, App은 RDB를 기본으로 사용
  * 여러 PC에서 사용하지만 Event 입력은 동시에 하지 않음
- App은 LDB import/export 및 자동 백업(local save) 기능 있음
  * LDB 자동 백업 시점: Event 발생 시, App 종료 시
  * LDB 백업 파일은 하루에 1개만 유지 (당일 백업 파일이 있으면 덮어씀)
  * 백업 파일명에 백업 날짜 포함 (예: LDB_backup_20260409.xlsx)
- LDB export: 자동 백업과 동일한 내용이나 사용자가 원하는 시점에 원하는 위치에 임의로 파일을 생성하는 기능. 자동 백업 파일과 별개로 관리됨
- LDB는 반드시 존재해야 함. LDB가 없는 상태로 local 모드 진입 시 데이터 없음으로 표시하여 사용자가 인지할 수 있도록 함
- RDB 초기화/리셋(마이그레이션) 기능: App install, patch 및 DB 조정에 따라 RDB는 사용자가 지정한 최신 LDB로 전체 대체될 수 있다.
  * 특정 App에서 DB 작업 등 시스템 점검 및 관리 중 다른 App에서 Event 입력 등의 사용자 조작은 없다.
  * RDB 초기화 전 기존 RDB 데이터 백업은 관리자가 App의 export 기능으로 수동 백업. 인적 오류 및 LDB 파일 손상은 설계 범위 아님
- DB 스키마 변경 시 관리자가 RDB와 LDB를 직접 수정한 후 배포. 이 작업 동안 App 및 각 DB는 사용하지 않음


[동작 모드]
- App 구동 시 RDB를 자동으로 가져와 적용(동기화)하고 remote 모드로 동작. 이때 App은 Event 발생 시마다 RDB에 기록
- App 구동 시 동기화에 실패하면(네트워크 단절, RDB 서비스 거부 등) local 모드로 동작. 이때 사용자가 LDB를 import하여 적용하고 Event 발생 시마다 LDB에 기록
- App이 remote 모드로 구동 중 동기화가 불가능해지면(네트워크 단절, RDB 서비스 거부 등) local 모드로 전환

[검토 필요] 모드 전환 시 데이터 유실 시나리오
  - remote→local 전환 시점에 마지막 RDB 기록 성공 이벤트와 전환 직전 발생 이벤트 사이의 간극 처리 방안
  - 미전송 이벤트 로컬 큐잉 및 이후 병합 시 반영 메커니즘 필요


[DB Event 병합]
- App이 구동되거나 모드가 변경되는 경우 RDB, LDB를 비교하여 Event 기록을 병합하고 RDB, LDB 모두 갱신하여 저장
- 병합 규칙:
  * 한쪽에만 존재하는 Event(ID 기준)는 무조건 추가
  * ID가 동일한 Event 간 충돌 발생 시 상태 기록 시각이 최신인 Event로 자동 대체 (사용자 판단 불필요)
  * ID가 동일하고 상태 기록 시각도 동일한 경우 LDB를 우선하여 적용
  * 충돌에서 선택되지 않은 Event는 버려짐

[검토 필요] RDB 동기화 방식
  - 구동 시: RDB 전체를 가져와 갱신하는 방식으로 동기화
  - 구동 이후: 변경분만 적용하는 방식으로 최적화 가능
  - 실제 구축 시 Firebase 관련 사항 추가 확인 및 검토 필요


[삭제 Event 영구 삭제 정책]
- 삭제(추가), 삭제(수정) 상태의 Event는 DB에 잔류하나 App에서 표시하지 않음
- App이 매일 자정에 삭제 상태 Event를 확인하여 상태 기록 시각 기준으로 7일이 경과한 Event는 RDB, LDB 모두에서 영구 삭제


[Event 구성]
- ID: Event 발행 정보로 구성되는 중복 불가 고유 ID
  - 형식: {장비명}_{발행시각}  (구분기호: 언더바 '_', 공백 허용 안 함)
  - 장비명: App이 구동 시 OS로부터 자동으로 획득 (PC 호스트명, 스마트폰/태블릿 기기명 등). 획득 실패 시 "unknown" 으로 처리
  - 발행시각: 날짜와 시각 정보 (예: 20260409110500)
  - 예시: MyLaptop_20260409110500, unknown_20260409110500

[검토 필요] 장비 식별 방식 고도화
  - 현재 방식(OS 자동 획득)의 잠재 이슈:
    1. 플랫폼 제한: iOS는 IDFV(앱 삭제 재설치 시 변경됨), Android는 ANDROID_ID(공장초기화 시 변경됨) 등 모바일 플랫폼은 영구적 하드웨어 ID 접근이 제한되어 획득값이 변경될 수 있음
    2. 중복 가능성: PC 호스트명은 사용자가 동일하게 설정할 수 있어 서로 다른 장비가 같은 장비명을 가질 수 있음
    3. 패치/재설치 후 장비명 불일치: 사용자가 장비명을 수동으로 재입력하는 방식으로 전환 시, 오타로 인해 동일 장비가 다른 장비로 인식될 수 있음
  - 개선 방향 후보: 장비 등록 정보를 RDB/LDB에 포함시키고 앱 설치 시 기존 목록에서 선택하는 방식으로 전환 검토

- Event 내용: Event 내용들 (이 설계에서 고려할 대상 아님)

- 상태 기록 시각: Event 상태가 생성되거나 변경된 시점의 날짜와 시각. 병합 시 충돌 자동 해결 기준 및 삭제 Event 영구 삭제 기준으로 사용됨

- Event 상태: 4가지
    추가됨      - 새로 Event가 추가된 상태
    수정됨      - Event가 수정된 상태
    삭제(추가)  - '추가됨' 상태에서 삭제된 Event. DB에 잔류하나 App에서 표시하지 않음
    삭제(수정)  - '수정됨' 상태에서 삭제된 Event. DB에 잔류하나 App에서 표시하지 않음

- Event 상태 전이 규칙:
    추가됨 → 수정됨      (추가된 Event 수정)
    추가됨 → 삭제(추가)  (추가됨 상태에서 삭제)
    수정됨 → 수정됨      (반복 수정)
    수정됨 → 삭제(수정)  (수정됨 상태에서 삭제)
    그 외 전이는 허용하지 않음 (삭제 상태에서의 추가 조작 불가)
